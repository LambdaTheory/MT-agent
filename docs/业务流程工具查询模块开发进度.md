## 1. 需求背景

用户要求把商品、链接、问题池、托管异常、筛选条件等查询统一成一个业务流程工具入口：

```text
LLM 只负责识别语义并产出结构化参数
→ 确定性工具执行查询/筛选
→ 统一展示卡片或文本
```

这个方向来自用户确认：托管异常、问题池、链接状态、筛选条件、本质都是同一类“链接/商品查询工作流程”，不应该继续为每种问法增加孤立 intent 或特殊 case。

必须保持的业务规则：

1. 普通 `查 703,706,736` 是批量商品查询，不是 ID 映射。
2. 普通 `查 733` 是商品查询。
3. 只有明确出现 `映射 / 转换 / 转 / 互查 / 对应 / 平台ID转端内ID / 端内ID转平台ID` 才走 ID 映射。
4. 商品展示必须同时显示：
   - `端内ID <id>`
   - `商品ID <platformProductId>`
   - 如果平台商品 ID 缺失，显示 `商品ID 未映射`，不能省略。
5. 单商品查询应返回详情卡，展示 1/7/30 日曝光、访问、发货。
6. 问题池卡片最多展示 5 条，并提供“查看完整清单”按钮。
7. 完整清单回调必须返回纯文本，并且只使用服务端生成的受控 queryRef，不能信任客户端传入任意筛选参数。

## 4. `productLink.query` 当前实现状态

当前 `src/feishuBot/productLinkQuery.ts` 的主要逻辑：

### 支持的 queryType

```ts
'productDetail'
'productList'
'problemPool'
'problemPoolCounts'
'sourceCoverage'
'linkStatus'
```

### 已实现路径

1. `problemPoolCounts`
   - 委托：
     ```ts
     runPublicTrafficReportQuery(context, { target: 'sectionCounts' })
     ```

2. `problemPool` / `linkStatus`
   - 构造 `target: 'section'` 的 reportQuery 参数。
   - 使用 `buildReportSectionCardData` 获取结构化 rows。
   - 使用 `queryProductResult` 转成双 ID 商品结果。
   - 使用 `buildProblemSectionCard` 返回问题池卡片。
   - metadata 包含：
     - `toolName`
     - `queryType`
     - `date`
     - `section`
     - `count`
     - `shownCount`
     - `productIds`
     - `queryRef`

3. `sourceCoverage`
   - 委托 `runPublicTrafficReportQuery(context, { target: 'sourceCoverage', ... })`。

4. `productList`
   - 如果没有明确 `productQuery`，或存在 filters/sortBy/limit/metrics，则委托：
     ```ts
     runPublicTrafficReportQuery(context, { target: 'products', ... })
     ```
   - 如果是明确多 ID，则继续走 `queryProductResult`，保证双 ID 展示。

5. `productDetail`
   - 使用 `queryProductResult(context, productQuery)`。
   - 单个 match 时返回 `buildProductDetailCard`。

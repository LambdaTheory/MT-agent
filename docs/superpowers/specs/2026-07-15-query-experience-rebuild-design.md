# 查询体验重构设计

**日期：** 2026-07-15  
**状态：** 已确认，待实施计划  
**范围：** 公域日报与 ID 映射快照上的只读商品、问题池和映射查询体验。

## 1. 背景与目标

当前只读查询存在三个相互关联的问题：

1. 问题池查询（例如“托管异常的链接有哪些”）只展示端内 ID，用户需要平台商品 ID 时仍得二次询问。
2. 用户复制一组端内 ID 后输入“查 703,706,736”可能误入单条 ID 映射能力，得到“没有找到 ID 映射”的错误语义，而非批量商品查询。
3. 单商品查询（例如“查 703”）沿用旧式简略文本，只显示 1 日和 7 日曝光、访问、发货；已有的双 ID、30 日指标和状态上下文没有被稳定展示。

本设计将“查”定义为**通用商品查询**：用户不必预先知道数字属于端内 ID 还是平台商品 ID。系统从最新可用的日报/映射快照中确定性识别，始终以双 ID 回显结果。只有用户明确表达“映射”或“转换”意图时，才进入专门的方向性 ID 映射入口。

本期目标：

- 让商品/问题池结果稳定显示 `端内ID` 与 `商品ID`（商品 ID 固定指平台商品 ID）。
- 让单 ID、批量 ID、混合 ID 的普通“查”都走同一个只读商品查询语义。
- 用两级飞书卡片提升问题池和单商品的可读性，同时为容量受限和卡片不可用情形提供字段等价的文本回退。
- 对未找到、映射缺失、数据源缺失和 ID 双重冲突显式说明，不能猜测或静默降级。

## 2. 非目标与边界

本期**不**：

- 发起实时 SaaS 商品读取，不展示价格、规格、实时库存或租期。
- 修改商品、链接、价格、规格、库存、活动或日报数据。
- 让问题池查询卡直接承载任何写操作。
- 将日报快照指标与实时数据混在同一张商品详情卡中。
- 因为普通批量“查”而强制扩展专用 ID 映射工具的批量转换能力。

所有结果只基于同一份最新可用公域日报、其状态快照和当前 ID 映射快照；卡片和文本必须标注报告日期、来源和更新时间。

## 3. 用户可见查询语义

### 3.1 三类入口

| 用户表达 | 路由 | 结果 |
| --- | --- | --- |
| `查 703`、`查 703,706,736`、`查 <平台商品ID>`、混合 ID 列表 | 通用商品查询 | 自动识别端内 ID/商品 ID，统一双 ID 回显；单条为详情，批量为紧凑结果。 |
| `端内ID 703 转商品ID`、`转换商品ID 703`、`商品ID映射 703` | 端内 ID → 商品 ID 映射 | 仅返回映射关系，不附加日报指标。 |
| `商品ID 2000 转端内ID`、`转换端内ID 2000` | 商品 ID → 端内 ID 映射 | 仅返回映射关系，不附加日报指标。 |

普通“查”不能因为输入是纯数字、逗号分隔或有多个数字而转入 ID 映射。

### 3.2 ID 识别、冲突与缺失

- 每个原始数字独立从报告行与 ID 映射快照解析端内 ID 与平台商品 ID。
- 唯一匹配一个商品时，自动返回该商品。
- 同一请求可包含端内 ID、商品 ID 和两者混合；已唯一命中的项正常返回。
- 如果同一数字同时作为端内 ID 和商品 ID 命中**不同商品**，系统不得猜测。返回澄清卡，列出两个候选的双 ID 与商品名，等待用户选择。
- 如果数字未在最新可用报告/映射快照中命中，集中列出未找到项，并明确“未在最新日报与 ID 映射快照中找到”；不能误称为“没有 ID 映射”。
- 找到商品但没有平台商品 ID 时，固定显示 `商品ID：未映射`；不得省略字段或虚构 ID。

### 3.3 字段与数据语义

- `端内ID <id>` 始终指内部商品 ID。
- `商品ID <id>` 始终指平台商品 ID。
- 普通商品查询的结果始终同时展示两个字段。
- 指标中数值 `0` 与来源缺失必须区分：可用来源的零值显示 `0`；来源不可用或指标不存在显示“暂无数据”或“来源不可用”。
- 不同来源/日期的数据不得混合；页脚说明报告日期、数据源与更新时间。

## 4. 结果呈现设计

### 4.1 模式选择

使用面向一线运营的 `ops_dashboard_card` 两级查询变体：

- 多商品问题池需要先扫描风险和对象，使用有容量上限的风险列表卡。
- 唯一商品需要核对身份、状态和窗口指标，使用信息层级更清晰的详情卡。
- 批量通用查询主要服务复制、复查和下一步输入，因此使用紧凑文本，而不是为每项发送详情卡。

这是只读状态同步设计，不是日报大卡、审批卡或写操作入口。

### 4.2 问题池/风险列表卡

适用于“托管异常的链接有哪些”“低曝光有哪些”等多商品问题池。

**卡头**

- 标题为 `<问题池标签> · <总条数>`，并显示日报日期。
- 高风险/异常使用红色卡头；普通无结果使用蓝色信息卡头。
- 颜色只表达状态：红色为异常/高风险，橙色为需关注但非高风险，蓝色为普通信息。不得将颜色作为装饰。

**首屏内容**

- 最多展示 5 条，采用纵向结构化列表，不使用移动端难读的宽表。
- 每条按以下顺序展示：
  1. `端内ID <id>｜商品ID <id 或 未映射>`；
  2. 商品名；
  3. 例如 `高优先级｜托管异常｜已托管 37 天` 的短状态行。
- 首屏不展示所有原始证据、长原因或更多报表指标。

**容量与完整清单**

- 结果超过 5 条时，在卡片内清楚写明未展示数量。
- 提供唯一主按钮：`查看完整清单`。
- 该按钮为只读回调：只返回当前受控查询结果/参数对应的完整纯文本清单，不信任回调中拼装的 ID、日期或筛选条件。
- 完整文本逐条使用与卡片相同的字段顺序，方便复制端内 ID 或商品 ID 后直接再次使用通用“查”。
- 没有更多结果时不展示按钮。

**无结果和回退**

- 无结果卡明确写“当前日报未发现 <问题池>”，并包含查询日报日期；不能渲染为空卡或当作系统错误。
- 卡片客户端不可用、卡片发送失败或结果不适合放入卡片时，回退为字段等价的纯文本；不能因回退丢失 ID、条目或状态。

### 4.3 单商品详情卡

适用于通用“查”唯一匹配一个商品，例如 `查 703`。

**卡头与身份**

- 使用蓝色信息卡头，标题为“商品查询结果”。
- 商品本身存在异常时，在正文状态标签中表达异常/优先级；普通查询卡整体不因数据异常而误用风险卡头。
- 首屏按顺序展示：商品名、`端内ID <id>｜商品ID <id 或 未映射>`、适用状态标签。
- 状态标签可包括 `托管异常`、`高优先级`、`已托管 N 天`；没有可靠状态时不虚构。

**窗口指标**

- 以纵向堆叠形式展示 `1 日`、`7 日`、`30 日`，每个窗口包含：`曝光`、`访问`、`发货`。
- 零值应保留；来源不可用时必须标为“暂无数据”或“来源不可用”，不得转写为零。
- 本期不展示价格、规格、库存、租期或其他实时 SaaS 信息。

**页脚**

- 固定展示：`数据源：公域日报｜报告日期：…｜更新时间：…`。
- 如存在映射缺失或来源缺失，在页脚/注释区域保留对应说明。

### 4.4 批量通用商品查询

适用于 `查 703,706,736` 或混合端内 ID/商品 ID 的输入。

- 默认使用紧凑纯文本，每条包含：
  `端内ID｜商品ID｜商品名｜1日/7日/30日：曝光、访问、发货`。
- 已命中项正常展示；未找到输入集中列出，不影响其他项。
- 有冲突输入时显示澄清卡；在用户选择前不得返回可能错误的商品行。
- 多条不逐条发详情卡，避免消息洪泛与卡片长度限制。

## 5. 飞书卡片交互与安全

### 5.1 交互原则

- 问题池卡仅有“查看完整清单”这一主操作。
- 卡片不提供任何写商品、重抓取、重建日报、发消息或改变业务状态的按钮。
- 单商品详情卡不需要交互按钮。
- ID 冲突使用独立澄清卡，候选项必须固定绑定到服务端确定的受控查询上下文。

### 5.2 回调安全与审计

- “查看完整清单”回调只依据服务端持久/短期存储的查询引用或重新验证过的确定性受控参数生成结果。
- 拒绝未知、过期、篡改、跨消息或无对应查询上下文的回调；失败时返回明确只读错误/状态，不读取任意客户端注入参数。
- SDK 长连接和 HTTP callback 必须调用同一份回调解析与查询逻辑，确保响应、回退和失败语义一致。
- 记录必要的只读运行日志（消息/查询引用、动作类型、成功/失败）；不得记录完整敏感业务数据、token 或凭据。

## 6. 技术边界与职责分配

| 层级 | 职责 |
| --- | --- |
| `src/agentData/` | 统一原始数字列表解析、端内/商品 ID 识别、冲突和未命中建模；形成双 ID、报告指标、来源状态统一结果模型。这里是计算和数据语义的唯一事实源。 |
| `src/feishuBot/` | 普通“查”的路由、明确映射/转换的专用路由、文本格式、问题池卡、单详情卡、完整清单回调及 HTTP/SDK 共享适配。不得重复计算 ID 识别或指标。 |
| `src/agentRuntime/` | 更新可见只读工具的 schema、metadata、工具描述与 planner grounding，明确普通查 ID 为商品查询，映射/转换为专用工具；保持执行端和 schema 同一契约。 |
| `tests/` | 覆盖数据层、意图/路由、工具契约、文本、卡片、回调、回退、SDK/HTTP 一致性与 planner 语义。 |

实施时优先扩展现有查询数据和格式化能力，而不是在大型执行器中继续堆叠业务计算。

## 7. 兼容策略

- 保留既有端内 ID、平台商品 ID、报告查询和 ID 映射工具的名称与已有显式语义。
- `查703`、`查 703,706` 等历史习惯继续有效，但目标改为通用商品查询，消除误入 ID 映射。
- 既有明确表达 `端内ID 703`、`平台商品ID 2000…` 继续支持，返回结果统一增加双 ID。
- “映射/转换”工具保持只回答映射关系；本期不改变高风险写工具对 ID 数组/单 ID 的严格参数边界。
- 卡片回退文本与卡片字段等价；回退不是减少信息的借口。

## 8. 验收与回归测试

### 8.1 数据与路由

- [ ] 单端内 ID 普通查询唯一匹配，返回双 ID 统一结果。
- [ ] 单平台商品 ID 普通查询唯一匹配，返回同一双 ID 统一结果。
- [ ] 批量端内 ID 普通查询不调用 ID 映射工具，正常部分成功。
- [ ] 混合端内 ID/商品 ID 输入可独立识别各项。
- [ ] 普通查询不存在的数字集中返回“最新日报与 ID 映射快照未找到”。
- [ ] 映射/转换显式词才路由至方向性映射工具，且结果不附加日报指标。
- [ ] 同一数字命中两个不同商品时返回澄清卡，未选择前不查询任一候选。
- [ ] 找到商品但无平台 ID 时显示 `商品ID：未映射`。

### 8.2 呈现

- [ ] `托管异常的链接有哪些` 卡片含日报日期、总数、双 ID、商品名、优先级、托管状态/时长。
- [ ] 问题池超过 5 条仅在卡中显示前 5 条，明确剩余数并显示“查看完整清单”。
- [ ] 完整清单回调返回全部条目和字段等价的纯文本。
- [ ] 问题池无结果显示信息状态与日期，不返回空卡。
- [ ] `查703` 返回详情卡，包含双 ID、名称、适用状态及 1/7/30 日曝光/访问/发货。
- [ ] 详情卡区分真实零值和来源缺失。
- [ ] 批量普通查询每项包含双 ID和 1/7/30 日核心指标。
- [ ] 卡片投递失败时自动回退为字段等价文本。

### 8.3 契约、回调与入口

- [ ] 更新的工具 schema、metadata、执行端、错误语义和 planner 提示一致。
- [ ] 普通“查 ID”在规划器可用和不可用的受控路径中都不会退化为映射查询。
- [ ] 完整清单回调拒绝未知、篡改、过期或跨消息的引用。
- [ ] HTTP callback 与 SDK callback 对有效、无效及回退场景的结果一致。
- [ ] 现有 `reportQuery`、`product.query`、`productId.lookup` 相关回归继续通过。
- [ ] 运行相关聚焦测试和 `npm run build`。

## 9. 实施完成标准

功能完成时，运营人员能直接复制问题池结果中的任一端内 ID 或商品 ID 继续使用“查”；系统无须用户提前区分 ID 类型。普通查询稳定双 ID 回显，显式映射仍然单一职责，卡片的容量限制或投递失败不会导致结果不完整或语义错误。

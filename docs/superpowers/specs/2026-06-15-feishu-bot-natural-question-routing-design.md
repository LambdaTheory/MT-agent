# 飞书机器人自然问句路由设计

## 背景

当前飞书机器人主要依赖精确命令，例如 `今日概况`、`查询 565`、`推送日报到群`。实际使用中，用户更常用自然问句，例如 `今天咋样`、`帮我看下新链接池`、`哪些链接不太健康`。这些问法容易落到 unknown，最终回复“不支持”。

项目已经具备只读数据查询能力：日报上下文读取、商品查询、任务池、问题商品、下架链接等。下一步应先扩大只读问询的理解范围，而不是直接引入 LLM。

## 目标

- 将机器人从“短命令匹配”扩展为“自然问句路由”。
- 优先覆盖只读查询类问法。
- 补齐已有 Agent intent 的工具接线：`overview`、`product`、`new_product_pool`、`order_summary`。
- 对不确定问题给出可用问法提示，而不是简单“不支持”。
- 保持会触发副作用的动作严格匹配，避免误跑日报、误推群。

## 非目标

- 不接入 LLM provider。
- 不实现多轮对话记忆。
- 不新增自动执行、审批、商品操作能力。
- 不改变群聊 @ 机器人后才响应的规则。

## 路由原则

机器人路由分为两类：

1. 只读查询类
   - 可使用更宽松的关键词、同义词、口语表达。
   - 误判成本较低，可以返回相关数据或空结果。

2. 副作用动作类
   - 保持严格命令。
   - 包括跑日报、重发日报、推送日报到群。
   - 不通过模糊语义猜测触发。

## 只读问询覆盖

### 今日概况

示例：

- `今天咋样`
- `今天数据如何`
- `现在公域怎么样`
- `日报概况`

路由到最新概况输出，返回最新日报总体摘要。实现上可以让 Agent `overview` 复用现有 `latest_summary` 格式。

### 商品查询

示例：

- `查一下721`
- `721怎么样`
- `帮我看下 Pocket 3`
- `这个商品 721 数据如何`

路由到商品查询。关键词可以是端内 ID、平台 ID 或商品名片段。

### 新链接池 / 冷启动

示例：

- `新链接池怎么样`
- `新链有哪些`
- `冷启动链接情况`
- `新品池现在有多少`

路由到 `new_product_pool`，返回新链接池摘要和主要明细。

### 待处理任务

示例：

- `有哪些要处理`
- `今天优先处理什么`
- `帮我列一下任务`

路由到 `tasks`。

### 问题商品

示例：

- `哪些转化差`
- `成交少的有哪些`
- `曝光低的链接`
- `不健康的链接`
- `哪些可以继续放量`

路由到对应 `problem_products` 类型。

其中 `不健康` 优先映射到综合待处理任务或问题商品集合，不触发执行动作。

### 下架链接

示例：

- `最近下架了哪些`
- `消失链接`
- `移除链接情况`

路由到 `removed_links`。

### 订单概况

示例：

- `订单情况`
- `发货怎么样`
- `履约情况`
- `归还关单情况`

路由到 `order_summary`。

## 未识别响应

当只读路由无法判断时，不再只回复固定“不支持”。改为提示可查询范围：

`我现在可以查：今日概况、商品、新链接池、待处理任务、转化差、曝光低、高潜力、下架链接、订单情况。你可以问“新链接池怎么样”或“查一下721”。`

## 数据流

1. `dispatcher` 继续负责群聊 @ 过滤和去重。
2. `parseBotIntent` 继续识别严格命令和直接商品查询。
3. unknown 文本进入 `parseAgentDataIntent`。
4. `handleBotIntent` 对 Agent intent 做统一工具调用。
5. 所有只读工具都从最新日报上下文读取数据。

## 安全边界

- `run_public_traffic_report` 必须包含明确的 `跑/生成/执行` 和 `日报` 语义。
- `push_latest_report_to_group` 只接受明确 `推送日报到群` 类语义。
- `resend_latest_report` 必须包含明确 `重发` 和 `日报`。
- 自然问句不能触发浏览器、扫码、抓取、推群。

## 测试要求

- 覆盖常见自然问句到只读 intent 的映射。
- 覆盖已有 Agent intent 在 `tools.ts` 中都能返回结果或合理空态。
- 覆盖模糊问句不会触发跑日报、重发、推群。
- 覆盖未识别响应包含可用问法提示。
- 保留现有精确命令兼容性。
